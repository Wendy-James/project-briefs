# Writing Guidelines

## Principles

- Be specific about the problem, scope, role, and constraints.
- Separate public facts from private implementation details.
- Prefer "designed", "explored", "analyzed", and "documented" when implementation evidence is not public.
- Include limitations and future work.
- Avoid unsupported metrics, polished marketing language, or claims that cannot be verified.

## Recommended Case Study Flow

1. Summary
2. Context
3. My Role
4. Problem Definition
5. Approach
6. Deliverables
7. Disclosure Notes
8. Lessons Learned
9. Future Work

## Useful Evidence Types

- Public repository links
- Public reports or presentations
- Architecture diagrams without confidential details
- Interface contracts
- Test strategy descriptions
- Clear lists of what is not included
